\section{Introduzione}\label{introduzione}

\subsection{Scopo del documento}\label{scopo-del-documento}

Il presente documento, denominato \textbf{Analisi dei Requisiti}, ha
l'obiettivo di esporre in modo dettagliato e non ambiguo
i requisiti funzionali, qualitativi e di vincolo individuati per il
progetto \textbf{Code Guardian}. Tali requisiti sono stati delineati a
seguito di un'attenta analisi del capitolato e dei
successivi incontri con il proponente. Il documento si avvale
dell'uso di diagrammi e descrizioni dei \textbf{Casi
d'Uso (UC)} per illustrare le interazioni tra gli attori
esterni e il sistema, fornendo una base solida per le successive fasi di
progettazione, codifica e verifica.

\subsection{Glossario}\label{glossario}

Per evitare ambiguità terminologiche e garantire una comprensione
univoca del documento, i termini tecnici, gli acronimi e le parole che
assumono un significato specifico nel contesto del progetto sono
riportati nel documento \emph{Glossario}. Questi termini sono
contrassegnati nel presente documento da una "G" a pedice ed è un
collegamento ipertestuale che rimanda direttamente alla definizione
corrispondente nel \emph{Glossario}. Tali parole vengono contrassegnate
solamente alla loro prima occorrenza in questo documento.

\section{Descrizione generale}\label{descrizione-generale}

\subsection{Obiettivi del prodotto}\label{obiettivi-del-prodotto}

L'obiettivo principale del progetto è lo sviluppo di un
set di agenti intelligenti basati su modelli linguistici di grandi
dimensioni (\emph{LLM}), progettati per integrarsi in
un'architettura di orchestrazione già esistente fornita
dal proponente \emph{Var Group}. Il sistema, denominato \textbf{Code
Guardian}, mira ad automatizzare i processi di audit, manutenzione e
documentazione dei repository software, riducendo il carico di lavoro
manuale dei team di sviluppo e migliorando la qualità complessiva del
codice.

Nello specifico, il prodotto si prefigge di:

\begin{itemize}
\item
  \textbf{Garantire la coerenza documentale}: assicurando che il README
  di progetto e la documentazione tecnica (inline e API) siano
  costantemente allineati all'evoluzione funzionale
  della piattaforma.
\item
  \textbf{Elevare gli standard di sicurezza}: introducendo il paradigma
  di \emph{policy-as-code}, attraverso il quale gli agenti verificano
  non solo le vulnerabilità generiche OWASP, ma anche
  l'aderenza alle specifiche pratiche di sviluppo sicuro
  definite dal team.
\item
  \textbf{Migliorare la trasparenza verso il business}: automatizzando
  la redazione di changelog comprensibili al cliente finale, traducendo
  i progressi tecnici emersi dalle user stories in valore di business
  tangibile.
\end{itemize}

\subsection{Funzioni del prodotto}\label{funzioni-del-prodotto}

Il prodotto offre le sue funzionalità attraverso tre agenti
specializzati coordinati dall'orchestratore centrale:

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\item
  \textbf{Agente di Documentazione e README}:

  \begin{itemize}
  \item
    \textbf{Manutenzione README}: creazione e aggiornamento dinamico del
    file README di progetto in base alle modifiche apportate alla
    piattaforma.
  \item
    \textbf{Gestione documentazione tecnica}: generazione e
    aggiornamento di documentazione inline (\emph{docstring},
    \emph{JSDoc}) e delle \emph{API docs} per riflettere lo stato
    attuale del codice.
  \end{itemize}
\item
  \textbf{Agente di Security Review (Policy-as-Code)}:

  \begin{itemize}
  \item
    \textbf{Analisi vulnerabilità}: utilizzo delle API di \emph{Claude
    Code} per l'identificazione di vulnerabilità
    standard \emph{OWASP}.
  \item
    \textbf{Verifica policy interne}: controllo
    dell'adozione di pratiche di sviluppo sicuro
    specifiche del team, utilizzando i file di configurazione
    \emph{CLAUDE.md} presenti nel repository come contesto operativo.
  \end{itemize}
\item
  \textbf{Agente di Changelog da User Stories}:

  \begin{itemize}
  \item
    \textbf{Analisi Sprint}: lettura e filtraggio delle \emph{user
    stories} completate durante uno specifico sprint di sviluppo.
  \item
    \textbf{Traduzione Business-Oriented:} conversione del linguaggio
    tecnico delle task in un linguaggio chiaro e comprensibile per il
    cliente, producendo \emph{changelog} sia tecnici che di business.
  \end{itemize}
\end{enumerate}

\subsection{Caratteristiche degli
utenti}\label{caratteristiche-degli-utenti}

Il sistema è progettato per inserirsi in un flusso di lavoro di sviluppo
software moderno, interagendo con diverse figure professionali che
operano all'interno di un team IT o nel rapporto con il
cliente finale.

L'utente principale è rappresentato dallo
\textbf{sviluppatore}, il quale interagisce con il sistema
indirettamente attraverso il repository Git. Egli beneficia
dell'automatizzazione della documentazione tecnica e del
feedback immediato sulla sicurezza, che gli permette di correggere
eventuali vulnerabilità o difformità rispetto alle policy aziendali in
tempo reale, senza attendere revisioni manuali.

Una seconda figura chiave è il \textbf{Team Lead} o lo \textbf{Security
Auditor}. Questi utenti utilizzano il sistema come strumento di
monitoraggio e governance: definiscono le regole di "sviluppo sicuro"
all'interno dei file CLAUDE.md e delegano
all'agente di sicurezza il compito di verificarne
l'adozione costante su tutto il codice prodotto.

Infine, il sistema si rivolge agli \textbf{Stakeholder di business}
(come Project Manager o il cliente finale). Questi soggetti non
necessitano di competenze tecniche approfondite, ma hanno bisogno di
visibilità sull'andamento del progetto. Per loro,
l'agente di changelog funge da "traduttore",
trasformando il gergo tecnico delle user stories in report di valore che
sintetizzano i progressi funzionali raggiunti al termine di ogni sprint.

\subsection{Piattaforma di esecuzione
{[}?{]}}\label{piattaforma-di-esecuzione}

Il prodotto è concepito come un ecosistema di agenti integrati in
un'infrastruttura già esistente, operante principalmente
in ambiente \textbf{cloud AWS}. Essendo il sistema basato su interfacce
di programmazione e automazione, l'esecuzione è
garantita attraverso i principali ambienti di runtime come
\textbf{Node.js} e \textbf{Python}. L'interazione con il
codice sorgente avviene tramite la piattaforma \textbf{GitHub},
sfruttando le \textbf{GitHub Actions} come principale motore di
esecuzione per l'analisi e
l'aggiornamento automatico dei repository.

\subsubsection{UC 1: Visualizzazione report prodotto dall'agente}
[descrizione]

\begin{itemize}
    \item \textbf{Attore Primario}: Sviluppatore.
    \item \textbf{Attori Secondari}: Orchestratore.
    \item \textbf{Trigger}: Lo Sviluppatore vuole consultare l'esito dell'analisi svolta da un agente.
    \item \textbf{Precondizioni}: L'Orchestratore ha eseguito almeno un agente. È disponibile almeno un report prodotto dall'agente.
    \item \textbf{Scenario Principale}:
        \begin{enumerate}
            \item Lo Sviluppatore accede alla sezione dedicata ai risultati dell'analisi.
            \item Il sistema mostra la lista dei report disponibili.
            \item Lo Sviluppatore seleziona il report desiderato.
            \item Il sistema visualizza il contenuto del report, includendo eventuali criticità, suggerimenti e file coinvolti.
        \end{enumerate}
    \item \textbf{Scenari alternativi:} Se non sono presenti report, il sistema mostra un messaggio informativo.
    \item \textbf{Postcondizioni}: Lo Sviluppatore visualizza il report generato.
\end{itemize}

\subsubsection{UC 2: Validazione umana di una proposta dell'agente}
Ad oggi per ogni sistema agentico è necessario implementare human-in-the-loop per assicurarsi i risultati affidabili. 

\begin{itemize}
    \item \textbf{Attore Primario}: Sviluppatore.
    \item \textbf{Attori Secondari}: Repository Git.
    \item \textbf{Trigger}: Lo Sviluppatore riceve una proposta generata da un agente.
    \item \textbf{Precondizioni}: Un agente ha prodotto una proposta di modifica. La proposta è disponibile sotto forma di Pull Request, patch o suggerimento testuale [da decidere...?]
    \item \textbf{Scenario Principale}:
        \begin{enumerate}
            \item Lo Sviluppatore apre la proposta generata dall'agente.
            \item Lo Sviluppatore analizza il contenuto dalla proposta.
            \item Lo Sviluppatore decide se accettare, modificare o rifiutare la proposta.
            \item Il sistema registra l'esito della validazione.
        \end{enumerate}
    \item \textbf{Postcondizioni}: La proposta è stata accettata, modificata o rifiutata dallo Sviluppatore.
\end{itemize}

\subsubsection{UC 3: Visualizzazione errore durante l’esecuzione dell’agente}
[descrizione]

\begin{itemize}
    \item \textbf{Attore Primario}: Sviluppatore.
    \item \textbf{Attori Secondari}: Orchestratore.
    \item \textbf{Trigger}: L'esecuzione di un agente termina con errore.
    \item \textbf{Precondizioni}: È stata avviata l'esecuzione di un agente. Si verifica un errore durante l'analisi o la generazione dell'output.
    \item \textbf{Scenario Principale}:
        \begin{enumerate}
            \item L'agente incontra un errore durante l'esecuzione.

            \item Il sistema indica, se possibile, un'azione correttiva.
        \end{enumerate}
    \item \textbf{Postcondizioni}: Lo Sviluppatore visualizza un messaggio di errore esplicativo.
\end{itemize}

\subsubsection{UC 4: Esportazione del report}
[descrizione]

\section{Agente 1: Documentazione e README}



\subsection{Introduzione all'agente}

L'Agente di Documentazione e README rappresenta un pilastro fondamentale per la qualità del lavoro quotidiano degli sviluppatori e per la stabilità del progetto Code Guardian. Sappiamo bene come, nei cicli di sviluppo veloci, la documentazione sia spesso la prima a perdere il passo rispetto al codice, diventando presto superata. Questo agente nasce proprio per risolvere tale problema: il suo compito è automatizzare tutto il processo di aggiornamento. In questo modo, garantiamo che il file README.md e l'intera documentazione tecnica, sia quella interna al codice che quella dedicata alle API, siano sempre uno specchio fedele e puntuale dell'evoluzione del progetto, in qualsiasi momento.



\subsection{Obiettivi e Funzionalità}

L'agente si focalizza su tre aree critiche:

\begin{enumerate}

    \item \textbf{Manutenzione del README}: Automazione della creazione e dell'aggiornamento del file principale di presentazione del repository, adattandolo dinamicamente alle evoluzioni della piattaforma.

    \item \textbf{Gestione della documentazione inline}: Analisi semantica del codice per la generazione di \textit{docstring} e \textit{JSDoc}, assicurando che ogni modulo sia correttamente commentato.

    \item \textbf{Documentazione delle API}: Estrazione automatizzata degli endpoint per la creazione di guide tecniche per i consumatori del servizio.

\end{enumerate}


\subsection{Casi d'Uso}
Di seguito i casi d'uso dell'agente 1.
\subsubsection{UC 5: Gestione del README di Progetto}
Questo caso d'uso riguarda il mantenimento del file \texttt{README.md} principale per riflettere lo stato attuale del software.

\begin{itemize}
    \item \textbf{Attore Primario}: Sviluppatore.
    \item \textbf{Attori Secondari}: Orchestratore, Repository Git.
    \item \textbf{Trigger}: Lo Sviluppatore effettua un push di modifiche strutturali o richiede l'aggiornamento tramite l'Orchestratore.
    \item \textbf{Precondizioni}: L'Agente 1 ha i permessi di lettura/scrittura sul repository; esiste un file \texttt{README.md}.
    \item \textbf{Scenario Principale}:
        \begin{enumerate}
            \item L'Orchestratore notifica l'Agente 1 della necessità di aggiornamento.
            \item L'Agente analizza file di configurazione (es. \texttt{package.json}) e struttura cartelle.
            \item L'Agente identifica nuove dipendenze o cambi di versione.
            \item L'Agente rigenera le sezioni interessate (es. guida all'installazione).
            \item L'Agente propone la modifica tramite Pull Request.
        \end{enumerate}
    \item \textbf{Postcondizioni}: Il file \texttt{README.md} è aggiornato e coerente con l'ultima versione del codice.
\end{itemize}

\subsubsection{UC 6: Generazione Documentazione Inline (Docstring/JSDoc)}
L'agente interviene nel codice sorgente per garantire che ogni componente sia documentato correttamente.

\begin{itemize}
    \item \textbf{Attore Primario}: Sviluppatore.
    \item \textbf{Attori Secondari}: Orchestratore.
    \item \textbf{Trigger}: Rilevamento di una funzione o classe priva di documentazione o con commenti obsoleti.
    \item \textbf{Scenario Principale}:
        \begin{enumerate}
            \item L'Agente interpreta la logica della funzione, i parametri di input e il valore di ritorno.
            \item L'Agente genera il blocco di commento standardizzato (JSDoc per JS/TS o Docstring per Python).
            \item L'Agente inserisce il commento sopra la firma della funzione.
        \end{enumerate}
    \item \textbf{Sottocasi}:
        \begin{itemize}
            \item \textbf{UC 2.1}: Correzione di documentazione esistente non più coerente (es. parametri cambiati).
            \item \textbf{UC 2.2}: Segnalazione di logiche troppo complesse per la documentazione automatica.
        \end{itemize}
\end{itemize}

\subsubsection{UC 7: Generazione Documentazione API}
L'agente estrae le definizioni degli endpoint per creare una guida tecnica per i consumatori delle API.

\begin{itemize}
    \item \textbf{Attore Primario}: Sviluppatore.
    \item \textbf{Trigger}: Richiesta esplicita di generazione prima di un rilascio o rilevamento nuove rotte.
    \item \textbf{Scenario Principale}:
        \begin{enumerate}
            \item L'Agente scansiona i file dei controller e i file delle rotte.
            \item Estrazione di: metodo HTTP, URL, header, payload richiesto e risposte possibili.
            \item Formattazione delle informazioni nel file \texttt{API.md} in formato Markdown.
            \item Salvataggio del documento nella cartella di documentazione predefinita.
        \end{enumerate}
    \item \textbf{Postcondizioni}: Disponibilità di un documento tecnico (\texttt{API.md}) aggiornato per gli stakeholder.
\end{itemize}




\subsection{Requisiti Funzionali}

I requisiti funzionali per l'Agente 1 sono definiti per rispondere alle esigenze di coerenza documentale espresse dal proponente:



\begin{itemize}

    \item \textbf{RF.1.1 (Aggiornamento README)}: Il sistema deve identificare le modifiche strutturali nel repository (es. aggiunta di moduli) e aggiornare le sezioni relative all'installazione nel \texttt{README.md}.

    \item \textbf{RF.1.2 (Standardizzazione Inline)}: Il sistema deve analizzare le funzioni prive di documentazione e generare commenti standardizzati che descrivano parametri di input, output e comportamenti attesi.

    \item \textbf{RF.1.3 (Workflow di validazione)}: Tutte le modifiche documentali proposte dall'agente devono essere sottomesse tramite \textit{Pull Request}, subordinando l'applicazione delle stesse a una revisione umana obbligatoria per garantire l'accuratezza del contenuto.

    \item \textbf{RF.1.4 (Documentazione API)}: Il sistema deve scandire i \textit{controller} dell'applicazione per produrre o aggiornare una guida Markdown (es. \texttt{API.md}) che elenchi metodi HTTP, rotte e payload richiesti.

\end{itemize}

\section{Agente 3: Scansione Vulnerabilità OWASP}

\subsection{Introduzione all'agente}
L'Agente di Scansione Vulnerabilità OWASP ha l'obiettivo di supportare lo sviluppatore nell'individuazione di possibili vulnerabilità di sicurezza presenti nel codice sorgente del repository. L'analisi di sicurezza viene svolta facendo riferimento alle principali categorie di rischio definite da OWASP individuabili tramite analisi del codice, come: injection, gestione non sicura dell'autenticazione, controllo degli accessi incompleto, gestione errata di dati sensibili, cofigurazioni insicure, ecc. \\ 
L'agente non garantisce la copertura esaustiva dell'intera OWASP Top 10. Il suo scopo è fornire una revisione automatizzata assistita, producendo segnalazioni, priorità e suggerimenti di remediation che devono essere successivamente valutati da uno sviluppatore.

\subsection{Obiettivi e Funzionalità}
L'agente si concentra su tre aree principali:
\begin{enumerate}
\item \textbf{Analisi delle vulnerabilità OWASP}: individuazione di pattern di codice potenzialmente riconducibili a vulnerabilità note, come SQL injection, uso insicuro di funzioni crittografiche, esposizione di informazioni sensibili, hardcoded secrets, ecc.
\item \textbf{Verifica delle policy interne di sicurezza}: controllo dell'aderenza del codice alle regole definite dal team nel file CLAUDE.md, secondo un approccio di tipo policy-as-code.
\item \textbf{Generazione di remediation}: produzione di suggerimenti testuali o frammenti di codice [quale cosa??] correttivi per mitigare le vulnerabilità o le non conformità rilevate, senza applicare modifiche automatiche prima della validazione umana.
\end{enumerate}

\section{Agente 4: Changelog da User Stories}

\subsection{Introduzione all'agente}
L'Agente di Changelog da User Stories funge da mediatore semantico tra il progresso puramente tecnico del team di sviluppo e le necessità informative degli stakeholder di business. In un contesto Agile, comprendere l'impatto reale di uno Sprint può essere complesso per figure non tecniche se i resoconti si limitano a liste di commit o ticket criptici. Questo agente automatizza la traduzione delle attività concluse in report chiari, garantendo che ogni aggiornamento del software sia accompagnato da una narrazione del valore di business generato, senza però tralasciare il dettaglio tecnico necessario alla manutenzione interna.

\subsection{Obiettivi e Funzionalità}
L'agente si concentra sulla sintesi e trasformazione del progresso di progetto attraverso tre direttrici:
\begin{enumerate}
\item \textbf{Recupero e Analisi dei Dati}: Integrazione con i sistemi di task management (Jira, GitHub Issues) per l'estrazione automatizzata dei task completati.
\item \textbf{Sintesi Semantica Multitarget}: Generazione di report differenziati; uno tecnico (Changelog Tecnico) per il team di sviluppo e uno di valore (Changelog di Business) per i clienti e il management.
\item \textbf{Validazione e Qualità del Contesto}: Monitoraggio della qualità informativa dei task per prevenire la generazione di report ambigui o privi di valore.
\end{enumerate}

\subsection{Casi d'Uso}
Di seguito i casi d'uso dell'agente 4.

\subsubsection{UC 7: Recupero e Filtraggio User Stories}
Questo caso d'uso riguarda l'estrazione delle attività completate dai sistemi esterni per avviare il processo di analisi.
\begin{itemize}
\item \textbf{Attore Primario}: Sviluppatore / Project Manager.
\item \textbf{Attori Secondari}: Orchestratore, Sistema di Task Management (GitHub/Jira).
\item \textbf{Trigger}: Lo Sviluppatore richiede la generazione del report a conclusione di uno Sprint.
\item \textbf{Precondizioni}: L'Agente ha i permessi di lettura sulle API del sistema di tracking; le task sono in stato "Done".
\item \textbf{Scenario Principale}:
\begin{enumerate}
\item L'Agente interroga il sistema tramite API filtrando per Sprint ID e stato.
\item L'Agente valida la presenza di titoli e descrizioni nelle User Stories recuperate.
\item L'Agente organizza i dati separandoli per tipologia (bugfix, feature, task tecnici).
\end{enumerate}
\item \textbf{Sottocasi}:
\begin{itemize}
\item \textbf{UC 7.1}: Rilevamento di task con metadati insufficienti e segnalazione all'utente.
\item \textbf{UC 7.2}: Gestione delle ambiguità se l'AI rileva discrepanze tra titolo e contenuto tecnico.
\end{itemize}
\end{itemize}

\subsubsection{UC 8: Generazione Changelog Tecnico (Dev-to-Dev)}
L'agente produce un documento rivolto ai programmatori che riassume i cambiamenti a basso livello.
\begin{itemize}
\item \textbf{Attore Primario}: Sviluppatore.
\item \textbf{Attori Secondari}: Orchestratore.
\item \textbf{Precondizioni}: Set di User Stories validato e pronto.
\item \textbf{Scenario Principale}:
\begin{enumerate}
\item L'Agente aggrega le User Stories per area di impatto tecnico.
\item Estrazione dei requisiti tecnici soddisfatti e dei riferimenti (ID task).
\item Formattazione del testo in linguaggio tecnico asciutto in formato Markdown (\texttt{CHANGELOG.md}).
\end{enumerate}
\item \textbf{Post-condizioni}: Generazione del file \texttt{CHANGELOG.md} tecnico.
\end{itemize}

\subsubsection{UC 9: Generazione Changelog di Business (Dev-to-Client)}
L'agente esegue una traduzione semantica per trasformare il gergo tecnico in benefici tangibili.
\begin{itemize}
\item \textbf{Attore Primario}: Sviluppatore.
\item \textbf{Attori Secondari}: Orchestratore, Modello AI (LLM).
\item \textbf{Trigger}: Conclusione con successo dell'UC 7.
\item \textbf{Precondizioni}: Disponibilità dei dati dello Sprint
\item \textbf{Scenario Principale}:
\begin{enumerate}
\item L'Agente interpreta il valore delle User Stories per l'utente finale tramite LLM.
\item L'AI converte termini tecnici in benefici (es. "Migliorata velocità" invece di "Ottimizzazione indici").
\item Filtro dei dettagli tecnici irrilevanti per lo stakeholder non tecnico.
\item Generazione di una bozza di report (PDF o testo) pronta per la revisione.
\end{enumerate}
\item \textbf{Post-condizioni}: Documento orientato al business generato e pronto per la revisione.
\end{itemize}

\subsubsection{UC 10: Validazione Formato Output AI}
Questo caso d'uso gestisce le situazioni in cui il modello linguistico produce un output non conforme agli standard tecnici richiesti.
\begin{itemize}
\item \textbf{Attore Primario}: Sviluppatore.
\item \textbf{Attori Secondari}: Modello AI (LLM).
\item \textbf{Trigger}: L'output generato dall'AI presenta errori di formattazione (es. Markdown corrotto o JSON non valido).
\item \textbf{Precondizioni}: L'Agente 4 ha terminato la fase di generazione.
\item \textbf{Scenario Principale}:
\begin{enumerate}
\item L'Agente riceve la risposta dall'LLM e ne analizza la struttura sintattica.
\item Se vengono rilevati errori, l'Agente richiede automaticamente una correzione al modello (Self-Correction).
\item In caso di persistenza dell'errore, il sistema notifica lo Sviluppatore dell'impossibilità di generare il report nel formato richiesto.
\end{enumerate}
\item \textbf{Post-condizioni}: Il report finale rispetta gli standard di formattazione richiesti.
\end{itemize}

\subsubsection{UC 11: Esportazione Report Multiformato}
L'agente permette la distribuzione dei report approvati in diversi formati a seconda del destinatario.
\begin{itemize}
\item \textbf{Attore Primario}: Project Manager / Sviluppatore.
\item \textbf{Trigger}: Approvazione manuale dei report generati.
\item \textbf{Precondizioni}: Report tecnico o business disponibile.
\item \textbf{Scenario Principale}:
\begin{enumerate}
\item L'utente seleziona il report approvato e sceglie il formato (PDF, Markdown, JSON).
\item Il sistema applica i template grafici o strutturali richiesti.
\item Il sistema genera il file per il download o il salvataggio automatico nel repository.
\end{enumerate}
\item \textbf{Post-condizioni}: Il file è correttamente scaricato.
\end{itemize}

\subsection{Requisiti Funzionali}
I requisiti funzionali per l'Agente 4 definiscono le capacità di analisi e trasformazione dei dati richiesti:

\begin{itemize}
\item \textbf{RF.4.1 (Integrazione Sistemi Task)}: Il sistema deve interfacciarsi con le API dei sistemi di Task Management esterni per il recupero dei metadati delle User Stories.
\item \textbf{RF.4.2 (Generazione Multitarget)}: Il sistema deve poter generare simultaneamente un report tecnico (orientato allo sviluppo) e un report di business (orientato al cliente).
\item \textbf{RF.4.3 (Traduzione Semantica)}: Il sistema deve utilizzare modelli di linguaggio (LLM) per mappare le attività tecniche a benefici funzionali descrittivi.
\item \textbf{RF.4.4 (Validazione Dati)}: Il sistema deve segnalare all'utente i task completati che presentano descrizioni ambigue o insufficienti per una corretta analisi automatizzata.
\end{itemize}




\section{Note Operative}

L'Agente 1 non è un software che richiede un intervento manuale costante. Esso opera in background: ogni volta che viene inviato nuovo codice al repository, l'agente riceve una notifica automatica. Invece di rielaborare l'intero progetto, esamina solo le parti che sono state effettivamente modificate. Questo permette di mantenere la documentazione aggiornata senza rallentare il flusso di lavoro degli sviluppatori, che ricevono solo le proposte di modifica finali pronte per essere revisionate.
